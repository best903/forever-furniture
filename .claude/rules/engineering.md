# 엔지니어링 가이드라인

## 운영 원칙
- **정확성 > 영리함**: 유지보수가 쉬운 평범하고 읽기 좋은 코드를 선호한다.
- **최소 변경 원칙**: blast radius를 최소화하고, 인접 코드 리팩토링은 위험/복잡도를 실질적으로 줄일 때만 수행한다.
- **기존 패턴 활용**: 새 추상화나 의존성 도입 전에 프로젝트의 기존 컨벤션을 따른다.
- **증명하라**: "맞는 것 같다"는 완료가 아니다. tdd.md의 "Test is Trust" 철학을 기반으로 하되, 테스트 외에도 빌드/lint/수동 검증 등 상황에 맞는 검증 수단을 포함한다.
- **불확실성을 명시하라**: 검증할 수 없으면 그렇다고 말하고, 가장 안전한 다음 단계를 제안한다.

## 점진적 전달 (Incremental Delivery)
- 큰 변경보다 **얇은 수직 슬라이스(thin vertical slice)**를 선호한다.
- 구현 → 테스트 → 검증 → 확장 순서로 작은 단위씩 진행한다.
- 가능하면 feature flag, config switch, 안전한 기본값 뒤에 변경사항을 둔다.

## 자기 개선 루프 (Self-Improvement Loop)
- Director의 교정이나 실수 발견 시, 프로젝트 루트의 `LESSONS.md`에 기록한다:
  - **실패 모드**: 무엇이 잘못되었는가
  - **감지 신호**: 어떻게 발견/감지했는가
  - **방지 규칙**: 재발을 막기 위한 규칙
- 세션 시작 시 프로젝트 루트의 `LESSONS.md`를 Read 도구로 확인하고 참고한다.
- 주요 리팩토링 전에도 `LESSONS.md`를 검토한다.

## 단계별 검증 (Verification at Every Step)
- 작업 중 매 단계마다 변경 전/후 동작을 비교하고 수시로 검증한다.
- 자문한다: "시니어 엔지니어가 이 diff와 검증 과정을 승인할까?"
- (최종 완료 기준은 아래 "완료의 정의" 체크리스트를 따른다)

## Stop-the-Line 규칙
예상치 못한 상황 발생 시(테스트 실패, 빌드 에러, 동작 회귀):
1. 즉시 기능 추가를 **중단**한다
2. 증거를 **보존**한다 (에러 출력, 재현 절차)
3. 진단으로 돌아가 **재계획**한다

## 오류 분류 체크리스트 (Error Triage)
순서대로 수행한다:
1. **재현**: 테스트, 스크립트, 또는 최소 절차로 안정적으로 재현
2. **격리**: 어느 레이어의 문제인지 파악 (UI, API, DB, 네트워크, 빌드)
3. **축소**: 최소 실패 케이스로 줄이기
4. **근본 원인 수정**: 증상이 아닌 원인을 고친다
5. **회귀 방지**: 테스트 또는 불변 조건 검사 추가
6. **종단간 검증**: 원래 리포트에 대해 전체 검증

## 범위 확장 통제 (Scope Creep Control)
- 변경 중 더 깊은 문제를 발견하면: 정확성/안전성에 필요한 것만 수정한다.
- 나머지는 현재 작업을 확장하지 말고, TODO/이슈로 기록한다.

## 커뮤니케이션
- **결과와 영향 중심**으로 소통한다. 과정 나열이 아님.
- 구체적 아티팩트를 참조한다: 파일 경로, 명령어, 에러 메시지, 변경 내용.
- 승인이 필요한 행동은 Absolute Rules(제1/5법칙)에 따라 반드시 보고한다. 이미 승인된 범위 내의 세부 진행과정은 축약하여 보고하며, 범위 변경, 리스크 발견, 검증 실패, 의사결정 필요 시에만 체크포인트를 제공한다.

## 완료의 정의 (Definition of Done)
PR/커밋 전 최종 체크리스트:
- [ ] 동작이 수용 기준에 부합
- [ ] 테스트/lint/typecheck/빌드 통과 (또는 미실행 사유 문서화)
- [ ] 증거 없이 완료 표시하지 않는다 (테스트 결과, 로그, 수동 재현 중 최소 하나)
- [ ] 위험한 변경은 롤백/flag 전략 존재
- [ ] 기존 컨벤션을 따르고 읽기 쉬운 코드
- [ ] 검증 스토리 존재: "무엇이 변경되었고 + 어떻게 확인했는가"
